## Maven Plugins

Plugins are the heart of the Maven build system. They provide additional functionality that enables you to perform specific tasks during the project build lifecycle, such as compiling code, running tests, packaging applications, generating reports, and more.

### Types of Maven Plugins

**Native Plugins (Core Plugins)**: Provided by Apache Maven, they handle standard tasks in the build process.

**Third-Party Plugins**: Extend Maven's capabilities to support functionality not included by default, such as framework integrations or documentation generation.

### Phases and plugins

| Phase|Description|Plugin |
| --------  | -------|-------|
|**clean**  | Clean up directories generated by previous builds, such as target.  |	**maven-clean-plugin**|
|**compile**| Compiles the project source code (src/main/java) to bytecode.|	**maven-compiler-plugin**|
|**test**   | Run the project's unit tests.|	**maven-surefire-plugin**|
|**package**| Package the project into a file (for example, JAR or WAR).|	**maven-jar-plugin** (or specific)|
|**install**| Install the generated package to the local repository (~/.m2/repository).	|**maven-install-plugin**	|
|**deploy** | Deploy the package to a remote repository so others can use it.|**maven-deploy-plugin**|

We can also extend and overwrite a native plugin, we have to do this in the pom.xml file.

```xml
<build>
  <plugins>
    <plugin>
      <groupId>org.apache.maven.plugins</groupId>
      <artifactId>maven-compiler-plugin</artifactId>
      <version>3.8.1</version> 
      <configuration>
        <source>17</source>
        <target>17</target>
      </configuration>
    </plugin>
  </plugins>
</build>
```

In this example we have added a specific version of the plugin (``<version>3.8.1</version>``) and we have also configured the java version (``17``) with which it has to be compiled.

It is recommended to make an extension of the plugin since if we do not do it, Maven will take the default versions of the plugin and the default versions of source and target which are Java 1.5 or 1.6

### How External Plugins Hook into the Lifecycle

External plugins are integrated into the build by binding their goals to a **lifecycle phase**.

- Lifecycle runs phases in order (for example: `validate -> initialize -> compile -> test -> verify -> package -> install -> deploy`)
- Each phase executes all goals bound to that phase
- You can bind third-party plugin goals with `<executions>` + `<phase>`

### `executions`, `execution`, and `goals` (Quick Explanation)

When configuring a plugin, these elements control **when** and **what** Maven runs:

- `<executions>`: container for one or more execution blocks
- `<execution>`: one runnable block (you can give it an `<id>`)
- `<phase>`: lifecycle phase where this execution should run
- `<goals>` / `<goal>`: concrete plugin tasks to execute

Basic shape:

```xml
<plugin>
  <groupId>...</groupId>
  <artifactId>...</artifactId>
  <executions>
    <execution>
      <id>my-execution</id>
      <phase>package</phase>
      <goals>
        <goal>some-goal</goal>
      </goals>
    </execution>
  </executions>
</plugin>
```

Important:

- `goal` names and valid configuration parameters are **plugin-specific**.
- There is no universal list of goals/parameters for all plugins.
- Always check each plugin documentation (or `mvn help:describe`) to know supported goals and options.

Complete JaCoCo example (multiple goals in different phases):

```xml
<build>
  <plugins>
    <plugin>
      <groupId>org.jacoco</groupId>
      <artifactId>jacoco-maven-plugin</artifactId>
      <version>0.8.10</version>
      <executions>
        <execution>
          <id>attach-agent</id>
          <phase>initialize</phase>
          <goals>
            <goal>prepare-agent</goal>
          </goals>
        </execution>

        <execution>
          <id>coverage-report</id>
          <phase>verify</phase>
          <goals>
            <goal>report</goal>
          </goals>
        </execution>

        <execution>
          <id>coverage-check</id>
          <phase>verify</phase>
          <goals>
            <goal>check</goal>
          </goals>
          <configuration>
            <rules>
              <rule>
                <element>BUNDLE</element>
                <limits>
                  <limit>
                    <counter>LINE</counter>
                    <value>COVEREDRATIO</value>
                    <minimum>0.80</minimum>
                  </limit>
                </limits>
              </rule>
            </rules>
          </configuration>
        </execution>
      </executions>
    </plugin>
  </plugins>
</build>
```

What happens with `mvn verify`:

1. `initialize` phase runs `jacoco:prepare-agent`
2. tests run in `test` phase and produce coverage data
3. `verify` phase runs `jacoco:report` and `jacoco:check`

JaCoCo has many other goals (`prepare-agent-integration`, `report-integration`, `merge`, `dump`, etc.), but goal names and valid configuration depend on the plugin.

### Examples Third-Party Plugins

1) **Spring Boot Maven Plugin**
   - **Plugin**: `spring-boot-maven-plugin`
   - **Usage**: The Spring Boot Maven Plugin simplifies the process of building and running Spring Boot applications. It enables packaging applications as executable JARs and provides commands to run the application directly.
   - **Main Tasks**:
     - Package the application as an executable `JAR` with all its dependencies.
     - Provides the `spring-boot:run` goal for running the application without building the JAR.
   - **Configuration Example**
        ```xml
        <build>
            <plugins>
                <plugin>
                    <groupId>org.springframework.boot</groupId>
                    <artifactId>spring-boot-maven-plugin</artifactId>
                </plugin>
            </plugins>
        </build>
        ```

2) **JaCoCo Maven Plugin**
   - **Plugin**: `jacoco-maven-plugin`
   - **Usage**: The JaCoCo Maven Plugin generates code coverage reports for unit tests, providing insights into the tested portions of your codebase.
   - **Main Tasks**:
     - Instruments the code to measure coverage during test execution.
     - Generates coverage reports in multiple formats (HTML, XML, CSV).
   - **Configuration Example**
        ```xml
        <build>
            <plugins>
                <plugin>
                    <groupId>org.jacoco</groupId>
                    <artifactId>jacoco-maven-plugin</artifactId>
                    <version>0.8.10</version>
                    <executions>
                        <execution>
                            <goals>
                                <goal>prepare-agent</goal>
                                <goal>report</goal>
                            </goals>
                        </execution>
                    </executions>
                </plugin>
            </plugins>
        </build>

        ```

3) **MapStruct Maven Plugin**
   - **Plugin**: `mapstruct-maven-plugin`
   - **Usage**: The MapStruct Maven Plugin is used to generate boilerplate code for object mapping. It simplifies the conversion of DTOs, entities, and other object types.
   - **Main Tasks**:
     - Generates implementation classes for annotated mapping interfaces.
     - Provides fast, type-safe mappings between Java objects.
   - **Configuration Example**
        ```xml
        <build>
            <plugins>
                <plugin>
                    <groupId>org.apache.maven.plugins</groupId>
                    <artifactId>maven-compiler-plugin</artifactId>
                    <version>3.10.1</version>
                    <configuration>
                        <annotationProcessorPaths>
                            <path>
                                <groupId>org.mapstruct</groupId>
                                <artifactId>mapstruct-processor</artifactId>
                                <version>1.5.5.Final</version>
                            </path>
                        </annotationProcessorPaths>
                    </configuration>
                </plugin>
            </plugins>
        </build>
        ```
